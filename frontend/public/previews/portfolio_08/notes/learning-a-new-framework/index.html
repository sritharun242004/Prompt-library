On Learning a New Framework

November 30, 2022
  • learning

Every few months there's a new framework I'm supposed to know. Here's how I approach learning them without losing my mind.

Start With the Official Docs

Not a YouTube series. Not a course. Not someone's "ultimate guide". The official documentation.

Docs are often dense, but they're accurate and maintained. The people who built the framework wrote them (or approved them). Everything else is someone's interpretation of the docs.

Build the Smallest Thing That Is Actually Useful

Not the tutorial project. A tiny version of something I actually need. This forces me to encounter the real edges — the stuff the tutorial skips because it's complicated.

Read the Source for the Hard Parts

When something behaves unexpectedly, I look at the source. This sounds intimidating but most modern frameworks are well-organized and fairly readable. Understanding how something works at one level down makes the surface API make more sense.

Timebox the Learning Period

I give myself a week of active learning, then I start building. The rest of the learning happens while I'm building. This prevents the trap of endless tutorial consumption without ever making anything.

What I Skip

The ecosystem. Every framework has a zoo of companion libraries, state management solutions, and "recommended" patterns. I skip all of this initially and add things only when I have a concrete problem they solve.
