Understanding Closures in JavaScript

April 10, 20242 min read
  • javascript
  • fundamentals

Closures are one of those JavaScript concepts that trips up almost everyone at first. Once it clicks, though, you start seeing them everywhere — and you realize you've been using them all along without knowing it.

What is a Closure?

A closure is a function that remembers the variables from the scope in which it was created, even after that outer scope has finished executing.

Here's the simplest example:

function makeCounter() {
  let count = 0

  return function () {
    count++
    return count
  }
}

const counter = makeCounter()
console.log(counter()) // 1
console.log(counter()) // 2
console.log(counter()) // 3

The inner function "closes over" the count variable. Even after makeCounter returns, the inner function keeps a live reference to count.

Lexical Scope

Closures depend on lexical (static) scoping — a function's scope is determined by where it is defined, not where it is called.

const greeting = 'Hello'

function greet(name) {
  // greeting is accessible here via the outer scope
  return `${greeting}, ${name}!`
}

Why Does This Matter?

Closures let you create private state. The count variable is inaccessible from outside makeCounter — you can only interact with it through the returned function.

This pattern is fundamental to:

  • Event listeners that remember context
  • Factory functions
  • The Module Pattern (before ES modules)
  • Partial application and currying

A Practical Example

Here's a factory that generates unique IDs:

function createIdGenerator(prefix) {
  let id = 0

  return function () {
    id++
    return `${prefix}-${id}`
  }
}

const generateUserId = createIdGenerator('user')
const generatePostId = createIdGenerator('post')

console.log(generateUserId()) // user-1
console.log(generateUserId()) // user-2
console.log(generatePostId()) // post-1

Each call to createIdGenerator creates a separate closure. The two generators don't share state.

The Loop Gotcha

One of the most famous interview questions involves closures and loops:

// The broken version
for (var i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i)
  }, 100)
}
// Logs: 3, 3, 3

var is function-scoped, so all three callbacks share the same i. By the time they run, i is 3.

Fix it with let:

for (let i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i)
  }, 100)
}
// Logs: 0, 1, 2

let is block-scoped — each loop iteration gets its own i binding.

Summary

  • A closure is a function bundled with references to its surrounding state
  • Lexical scope determines which variables a function can see
  • Closures enable private state, factory functions, and event handlers
  • Watch the var-in-loops gotcha; prefer let or const
