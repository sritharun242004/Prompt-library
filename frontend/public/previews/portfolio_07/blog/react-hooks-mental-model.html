React Hooks: The Mental Model That Makes Everything Click

useEffect confuses almost every developer who encounters it. The reason is not complexity — it is a missing mental model. Once you understand hooks as synchronisation tools rather than lifecycle methods, everything makes sense.

December 20, 202412 min read

I have reviewed hundreds of React code samples over the years. The most common source of bugs — far ahead of state mutation, key prop mistakes, or async handling — is useEffect used incorrectly. And almost always, the root cause is the same: the developer is thinking about useEffect as a lifecycle method.

The Wrong Mental Model: Lifecycle Methods

In class components, lifecycle methods had clear meanings tied to component life: componentDidMount fires once when the component appears. componentDidUpdate fires after every update. componentWillUnmount fires before the component disappears.

When hooks were introduced, developers mapped the new APIs to the old mental model. useEffect with an empty dependency array became "componentDidMount." useEffect with a dependency array became "componentDidUpdate." useEffect with a cleanup function became "componentWillUnmount." This mapping works just often enough to seem right, and it causes bugs subtle enough to be hard to diagnose.

The Right Mental Model: Synchronisation

The correct way to think about useEffect is: this code synchronises some external system with the current value of these reactive values.

An effect that fetches data based on userId is synchronising the network with the current userId. An effect that sets up an event listener is synchronising the DOM event system with the current callback. An effect that starts a timer is synchronising the timer system with the current duration.

This reframe changes what you expect from useEffect. If userId changes, the synchronisation runs again — because the external system (the network) needs to be updated to match the new userId. Of course it re-runs. That is what synchronisation means.

Dependency Arrays Are Not Skip Lists

The most damaging misconception is that the dependency array is a performance optimisation — a way to skip running the effect on renders where you do not want it. This is wrong. The dependency array is a promise to React: "these are all the reactive values this effect uses."

If you omit a value from the dependency array that the effect actually reads, you get a stale closure — the effect sees an old version of the value even when the value has changed. This causes the class of bugs that makes developers say "but I checked and the value was right!"

ESLint's exhaustive-deps rule catches these mistakes. It is worth enabling and respecting, even when it seems to demand things that do not make sense at first. If the rule asks you to add a function to the dependency array, the right response is not to add an ESLint disable comment — it is to understand why the function changes identity on every render and fix that with useCallback.

The Event Handler Exception

Not everything that happens in a component needs to go in useEffect. Event handlers are not effects — they are responses to user intent. A button click handler that calls a function should be a regular function passed to onClick, not an effect that responds to a state change.

The confusion arises when developers want to do something after a state update. Because you cannot safely read updated state immediately after calling setState (the update is asynchronous), they reach for useEffect to observe the state change. In most cases, there is a cleaner way: compute the value before updating state, or restructure the logic to not require that sequencing.

useLayoutEffect for Synchronous DOM Reads

useEffect runs after the browser paints. useLayoutEffect runs synchronously after the DOM is mutated but before the browser paints. Use useLayoutEffect when you need to read layout information (element dimensions, scroll position) and immediately react to it — otherwise users see a flash of incorrect layout.

The rule of thumb: start with useEffect. If you see a flash or an inconsistent visual state, switch to useLayoutEffect. If you are server-side rendering, be aware that useLayoutEffect on the server generates a warning — it cannot run in a non-browser environment.

The Cleanup Function Is a Re-synchronisation Tool

When an effect returns a function, React runs that function before the effect runs again and before the component unmounts. This is not "clean up on unmount" — it is "stop the previous synchronisation before starting the next one."

An event listener effect adds the listener in the effect body and removes it in the cleanup. When the event type changes, React removes the old listener and adds the new one. This is exactly the right behaviour. The cleanup is not about component lifecycle; it is about keeping the synchronisation correct.

Understanding this makes the empty dependency array case obvious: cleanup runs once on unmount because there is only one synchronisation session. With dependencies, it runs every time the dependencies change — which is exactly when you need to re-synchronise.

Practical Debugging

When an effect is behaving unexpectedly, ask: "what external system am I synchronising, and what reactive values does that synchronisation depend on?" Trace through what happens when each dependency changes. If the answer is "nothing should happen," that is your signal that either the value should not be a dependency, or the effect is structured incorrectly.

The mental model of synchronisation is not a metaphor — it is literally how React's runtime works. Hooks are designed to express synchronisation logic. When you write with that intent, the dependency arrays become obvious, the bugs become fewer, and the code becomes easier to reason about.
