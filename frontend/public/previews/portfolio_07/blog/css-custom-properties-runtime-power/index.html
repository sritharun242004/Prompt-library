CSS Custom Properties: The Runtime Feature You Are Ignoring

CSS variables are not just theming shortcuts. They are live values computed at runtime, inherited through the DOM, and overridable at any scope. Understanding this unlocks an entirely new class of CSS techniques.

January 12, 20258 min read

Most developers discover CSS custom properties as a way to avoid repeating colour hex codes. --brand-blue: #2563EB, used everywhere a blue is needed. This is useful, but it is barely scratching the surface.

Custom properties are live values. They are inherited. They can be updated with JavaScript. They respond to media queries and container queries. They can hold any CSS value, not just colours. Understanding this changes what you reach for when solving CSS problems.

Custom Properties Are Inherited

When you set a custom property on an element, all its descendants inherit it — unless they override it. This is the same inheritance model as color and font-size. It means you can scope custom properties to components by setting them on the component root.

A card component can define its own --card-bg and --card-border, and all the nested elements use those values. When you need a dark version, set data-variant="dark" on the card element and override just those two properties. The card's children do not need to know about the variant at all.

Scoped Overrides Beat Modifier Classes

The old BEM way of handling component variants was to add modifier classes: card--dark, card--featured, card--compact. Each modifier class had to re-declare every property that changed. This created specificity issues and made the CSS hard to scan.

With custom properties, the variant only needs to override the tokens that change. The base component styles reference the tokens. A dark card changes --card-bg and --card-border; it does not re-declare the card's padding, border-radius, or typography.

This is how the design token approach works in practice. The tokens are the interface between the design system and the component implementation.

JavaScript Can Update Them

document.documentElement.style.setProperty('--accent', 'hsl(333deg 100% 65%)') updates a custom property in real time. This is how theme toggles work without reloading the page. Set a property on :root and every element that references it immediately recomputes.

The implications go beyond theming. You can drive CSS animations by updating a single number in JavaScript. A scroll-linked animation can set --scroll-progress to a value between 0 and 1, and the CSS animation uses that value via calc() to interpolate positions, opacity, or scale. This avoids the performance cost of setting inline styles on every frame.

@property: Typed Custom Properties

The @property at-rule lets you declare the type and initial value of a custom property. This has two major benefits. First, it enables transitions and animations on custom properties — by default, custom properties cannot be transitioned because the browser does not know their type. With @property, you can transition a custom property that holds a colour or a number.

Second, it documents your design tokens. A @property declaration that specifies syntax: '<color>' and initial-value: hsl(225deg 100% 75%) is self-documenting in a way that a comment never is.

The Fallback Mechanism

Custom properties have a fallback syntax: var(--color-primary, blue). If --color-primary is not defined in scope, the fallback value blue is used. This is useful for component APIs — define a custom property as the component's configuration interface, and provide a sensible default.

A Button component that accepts --button-bg as an override can use var(--button-bg, var(--color-primary)) as its background. Most of the time, it uses the theme's primary colour. When a specific button needs a different colour, the parent element sets --button-bg without touching the component's CSS.

Practical Takeaways

Adopt a naming convention for your tokens. I use --color-*, --space-*, --font-*, and --radius for global tokens, and component-specific tokens named after the component. Keep the global tokens in :root. Override them at component scope when needed.

Reach for custom properties before reaching for JavaScript when you need dynamic styles. In most cases, a JS value setting a CSS property is faster to implement and easier to maintain than a JS value that computes a style string.

Custom properties are one of the highest-leverage CSS features available today. Learning to think with them will change how you architect stylesheets.
