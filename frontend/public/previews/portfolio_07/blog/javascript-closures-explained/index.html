JavaScript Closures: The Mental Model That Unlocks React Hooks

Closures are the mechanism behind React hooks, module patterns, and every setTimeout callback that has ever surprised you. Once you see how JavaScript scopes work, you cannot unsee it.

December 5, 20249 min read

Every experienced JavaScript developer has been burned by a closure. Usually it happens inside a loop, or inside a setTimeout, or when a React effect captures a stale variable. The bug is hard to reproduce, the fix feels wrong, and the cause takes a while to understand.

Closures are not advanced JavaScript. They are a fundamental property of how the language works. Once the mental model is clear, you stop writing code that creates this class of bug, and you start writing code that deliberately uses closures to your advantage.

What Is a Closure?

A closure is a function that remembers the variables from the scope where it was created, even after that scope has finished executing.

When a function is defined inside another function, it has access to the outer function's variables. That access persists even after the outer function has returned. The inner function has "closed over" those variables — it carries a reference to them.

This is not a special feature you opt into. Every function in JavaScript is a closure. The question is just which variables it closes over and when those variables change.

The Loop Bug

The classic closure bug involves a loop and a timer. You write a loop that schedules a function for each iteration. You expect each function to remember the current index. They all remember the same index — the final one.

The reason: var is function-scoped, not block-scoped. The loop body does not create a new scope for each iteration. All the timer callbacks close over the same i variable, and by the time they run, the loop has finished and i holds its final value.

The fix with let creates a new binding for each iteration, so each closure captures a different variable. The fix with a closure wrapper calls an immediately-invoked function to create a new scope and pass the current value as a parameter.

Modern JavaScript with let and const largely eliminates this bug in loop contexts, but the underlying mechanism still trips developers up in React effects and event listeners.

Stale Closures in React

This is where closures become a daily concern for React developers. When you define a function inside a component, it closes over the component's state and props at the time it was created. If state updates, the function still references the old value — unless React re-creates the function.

The linter rule exhaustive-deps exists to catch exactly this. When you use a stale value in an effect, the effect does not see the update. When you use a stale value in an event handler memoised with useCallback, the handler silently operates on outdated data.

The most common manifestation: a useEffect that should run when a piece of state changes, but the developer does not add that state to the dependency array. The effect runs once and captures the initial state. The state updates, but the effect never re-runs, so it continues to operate on the initial value.

Closures as a Design Pattern

Once you understand closures, you can use them intentionally. The module pattern uses closures to create private state. A counter function that exposes increment, decrement, and value — but keeps the internal count variable inaccessible from outside — is a closure.

React hooks themselves are implemented with closures. The useState hook closes over the component's state in the React runtime. The hook can read and update that state because it has access to the closed-over reference, even though the hook itself is a plain function call.

Custom hooks are closures too. When you write useLocalStorage, it captures the key you pass it and closes over the useState setter. The hook "remembers" the key without you having to pass it again on every interaction.

Memoisation With Closures

Memoisation — caching the result of expensive function calls — is naturally expressed with closures. A memoised function closes over a cache object. On each call, it checks the cache for the given arguments. If a cached result exists, it returns it; if not, it computes the result, stores it in the cache, and returns it.

The cache persists between calls because it is closed over. It is not a global variable — it is private to the memoised function. This is clean, composable design.

Practical Takeaways

Treat every function you write as a potential closure. Ask: what variables does this function access? When were those variables bound? When might they change?

In React, pay attention to what your hooks and handlers capture. When the linter flags a missing dependency, take it seriously rather than suppressing the warning. The stale closure bug it is preventing is genuinely hard to debug.

Once closures are part of your mental model, you start seeing them everywhere — in event delegation, in module patterns, in promises, in iterators. JavaScript's entire async story is built on closures. The language rewards developers who understand this deeply.
