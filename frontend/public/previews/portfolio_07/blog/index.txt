## The Mental Model: Tracks and Cells

A grid is made of tracks — horizontal rows and vertical columns. Where tracks cross, you get cells. Items can span multiple cells to form areas. This sounds simple, but it fundamentally rewires how you approach layout problems.

The fr unit is the key insight: it means "fraction of available space after fixed tracks are removed." One 1fr column next to a 2fr column means 33% and 66% — but only after fixed-width columns and gaps are subtracted. This is subtly different from percentages, and the difference matters in real layouts.

## Named Grid Areas

This is where Grid gets magical. Name your cells, then place items into those names:

When you use grid-template-areas, your CSS becomes readable as a visual map of the layout. When you need a responsive layout, you just redefine the areas in a media query — no touching the child elements. The children keep their grid-area names and flow into whatever positions you specify.

I have refactored layouts that used to be 80 lines of flexbox and media queries into 15 lines of grid. That is not just efficiency — it is clarity. Clarity is what lets you come back to code six months later and immediately understand what it does.

## The Auto-Placement Algorithm

When you do not explicitly place items, Grid runs an auto-placement algorithm. Understanding it prevents layout surprises. The algorithm fills cells left-to-right, top-to-bottom, which matches the DOM order. If an item spans multiple columns, it jumps to the next row where it fits.

The dense keyword tells the algorithm to backfill gaps left by larger items. This is perfect for image galleries where item sizes vary. Without dense (the default is sparse), items always maintain document order even when gaps appear.

## Subgrid: Alignment Across Cards

Subgrid lets a child grid participate in the parent's track sizes. This solves the card alignment problem that plagued CSS for a decade. Imagine a three-column card grid where each card has a title, body text, and a footer button. Without subgrid, the footers sit at different heights because the body text lengths vary.

With subgrid, each card spans three row tracks from the parent grid. The title, body, and footer in every card align with the corresponding row tracks in every other card. The result feels like a design system working as intended.

## Responsive Without Media Queries

One of Grid's most underrated features is repeat(auto-fill, minmax(200px, 1fr)). This single declaration creates a responsive grid that adds and removes columns as the container width changes — no media queries required. The browser calculates how many 200px-minimum columns fit in the available space and creates exactly that many.

This pattern is so useful that I reach for it automatically when building any card grid. It handles everything from a single-column mobile view to a six-column desktop view without a line of breakpoint CSS.

CSS Grid is not just a feature — it is a new vocabulary for layout. When you internalize tracks, areas, and the fr unit, you stop fighting the browser and start collaborating with it. The layouts that feel impossible in other systems become almost trivially simple in Grid. ## How React Decides to Re-render

React re-renders a component whenever its state or props change. That sounds straightforward, but the details matter. When a parent re-renders, all its children re-render by default — even if their props have not changed. This is not a bug; it is by design. React's reconciler is fast enough that skipping the render phase usually costs more than just doing it.

This is the first thing most developers misunderstand: re-rendering is not the same as updating the DOM. React can render a component and produce a virtual DOM diff, discover nothing changed, and skip the DOM update entirely. The expensive part is the DOM update, not the render.

## When memo Actually Helps

React.memo and useMemo are frequently overused. They add complexity and have their own cost — the cost of comparing props on every render. For most components, this comparison is more expensive than just rendering.

memo genuinely helps in two cases. First, when a component renders a large tree and its props almost never change — like a sidebar that only updates on route changes. Second, when a component is inside a list that renders hundreds of items and each list item has stable props.

The trap is reaching for memo preemptively. Every time I have profiled a slow React app, the bottleneck has been something other than excessive re-renders. It has been a synchronous effect running on every keystroke, or a derived value being recalculated with an O(n²) loop, or a missing key prop causing the reconciler to unmount and remount components.

## The Key Prop Is Your Friend

Incorrect key props cause more performance problems than almost anything else. When you use an index as a key in a list that can be reordered or filtered, React has to diff every element on every change instead of simply moving DOM nodes around.

Use stable, unique identifiers as keys. If your data does not have natural IDs, generate them at ingestion time rather than at render time. This single change has eliminated rendering bottlenecks in more apps than I can count.

## Expensive Computations and useMemo

useMemo is worth reaching for when you have a genuinely expensive computation that depends on values that change rarely. Parsing a large dataset, computing a graph layout, filtering thousands of items — these are good candidates.

The key is to measure first. React DevTools Profiler shows you exactly where time is being spent. Without measurement, you are guessing, and guessing wrong adds complexity without improving performance.

## State Colocation

Moving state down the component tree is one of the highest-leverage optimisations available. When a piece of state only affects a subtree, it belongs in that subtree. State at the top of the tree causes the entire tree to re-render on every update.

A text input that lives in the root component will re-render your entire application on every keystroke. The same input colocated in the form component only re-renders the form. This is not an academic difference — it is often the difference between a 60fps experience and a sluggish one.

Optimise after you measure, not before. Use the Profiler. Look for renders that take more than 16ms. Look for components that render far more often than you expect. Fix the worst offenders first. The cubic-bezier function takes four numbers that define the control points of a Bezier curve. The curve maps time to progress — a steep early slope means fast initial motion, a gentle late slope means slow arrival.

Easing curves are approximations of physics. True spring physics — used by Framer Motion and React Spring — is computed frame by frame, which means the animation can be interrupted naturally. ## What Generics Actually Are

A generic is a type parameter — a placeholder that gets filled in when the type is used. It is the same idea as a function parameter, but for types instead of values.

Think about the Array type. An array of strings and an array of numbers are different things, but they share the same operations: push, pop, map, filter. Rather than duplicating the Array implementation for every possible element type, TypeScript uses a generic: Array<T>. When you write string[], TypeScript fills in T as string.

That is it. Generics are just reusable type slots.

## The Simplest Generic Function

The canonical example is an identity function — a function that returns whatever you pass it. Without generics, you would either type it as (x: any) => any, which loses all type information, or duplicate it for every type you need.

With a generic: function identity<T>(x: T): T { return x }. Now TypeScript knows that if you pass a string, you get a string back. If you pass a number, you get a number back. The type flows through the function.

## Constraints: T extends Something

The power comes when you constrain the generic. T extends SomeType means "T can be any type that is a subtype of SomeType." This lets you write functions that work on any object with certain properties.

A common real-world example: a function that picks a subset of keys from an object. You want it to work on any object, but you need to know the object's shape and the key names are valid. The constraint T extends object and K extends keyof T gives you exactly that.

## Generics in React

In React, generics show up constantly in typed component props. A generic List component that can render a list of any item type — as long as you tell it how to render each item and how to get a unique key — is far more reusable than a ListOfStrings component.

The pattern looks like: function List<T>({ items, renderItem, getKey }: ListProps<T>). The caller provides the type, either explicitly or through inference, and TypeScript ensures the renderItem function receives the right type.

## Conditional Types and Inference

Advanced generics include conditional types: T extends U ? X : Y. This sounds abstract, but the practical use is type unwrapping. If you have a function that returns a Promise<T>, you can write a utility type that extracts T — the type inside the promise.

The infer keyword inside conditional types lets TypeScript figure out part of a type automatically. It is how utility types like ReturnType and Awaited work in the standard library.

## When to Reach for Generics

A reliable signal that generics are the right tool: you have written almost the same type or function more than twice, and the only difference is a specific type. Extract the type into a parameter. The duplication disappears and the behaviour is identical.

A reliable signal that generics are the wrong tool: you are reaching for T because you want flexibility, but you will always call the function with the same concrete type. At that point, just use the concrete type. Generics add a layer of indirection that has a cost — in readability and in compile time.

## Custom Properties Are Inherited

When you set a custom property on an element, all its descendants inherit it — unless they override it. This is the same inheritance model as color and font-size. It means you can scope custom properties to components by setting them on the component root.

A card component can define its own --card-bg and --card-border, and all the nested elements use those values. When you need a dark version, set data-variant="dark" on the card element and override just those two properties. The card's children do not need to know about the variant at all.

## Scoped Overrides Beat Modifier Classes

The old BEM way of handling component variants was to add modifier classes: card--dark, card--featured, card--compact. Each modifier class had to re-declare every property that changed. This created specificity issues and made the CSS hard to scan.

With custom properties, the variant only needs to override the tokens that change. The base component styles reference the tokens. A dark card changes --card-bg and --card-border; it does not re-declare the card's padding, border-radius, or typography.

This is how the design token approach works in practice. The tokens are the interface between the design system and the component implementation.

## JavaScript Can Update Them

document.documentElement.style.setProperty('--accent', 'hsl(333deg 100% 65%)') updates a custom property in real time. This is how theme toggles work without reloading the page. Set a property on :root and every element that references it immediately recomputes.

The implications go beyond theming. You can drive CSS animations by updating a single number in JavaScript. A scroll-linked animation can set --scroll-progress to a value between 0 and 1, and the CSS animation uses that value via calc() to interpolate positions, opacity, or scale. The @property at-rule lets you declare the type and initial value of a custom property. This has two major benefits. First, it enables transitions and animations on custom properties — by default, custom properties cannot be transitioned because the browser does not know their type. Second, it documents your design tokens. A @property declaration that specifies syntax: '<color>' and initial-value: hsl(225deg 100% 75%) is self-documenting in a way that a comment never is.

Custom properties have a fallback syntax: var(--color-primary, blue). If --color-primary is not defined in scope, the fallback value blue is used. This is useful for component APIs — define a custom property as the component's configuration interface, and provide a sensible default.

In class components, lifecycle methods had clear meanings tied to component life: componentDidMount fires once when the component appears. componentDidUpdate fires after every update. componentWillUnmount fires before the component disappears.

When hooks were introduced, developers mapped the new APIs to the old mental model. useEffect with an empty dependency array became "componentDidMount." useEffect with a dependency array became "componentDidUpdate." useEffect with a cleanup function became "componentWillUnmount." ## The Right Mental Model: Synchronisation

The correct way to think about useEffect is: this code synchronises some external system with the current value of these reactive values.

An effect that fetches data based on userId is synchronising the network with the current userId. An effect that sets up an event listener is synchronising the DOM event system with the current callback. An effect that starts a timer is synchronising the timer system with the current duration.

This reframe changes what you expect from useEffect. If userId changes, the synchronisation runs again — because the external system (the network) needs to be updated to match the new userId. Of course it re-runs. That is what synchronisation means.

## Dependency Arrays Are Not Skip Lists

The most damaging misconception is that the dependency array is a performance optimisation — a way to skip running the effect on renders where you do not want it. This is wrong. The dependency array is a promise to React: "these are all the reactive values this effect uses."

## What Is a Closure?

A closure is a function that remembers the variables from the scope where it was created, even after that scope has finished executing.

When a function is defined inside another function, it has access to the outer function's variables. That access persists even after the outer function has returned. The inner function has "closed over" those variables — it carries a reference to them.

This is not a special feature you opt into. Every function in JavaScript is a closure. The question is just which variables it closes over and when those variables change.

## The Loop Bug

The classic closure bug involves a loop and a timer. You write a loop that schedules a function for each iteration. You expect each function to remember the current index. They all remember the same index — the final one.

The reason: var is function-scoped, not block-scoped. The loop body does not create a new scope for each iteration. All the timer callbacks close over the same i variable, and by the time they run, the loop has finished and i holds its final value.

The fix with let creates a new binding for each iteration, so each closure captures a different variable. The fix with a closure wrapper calls an immediately-invoked function to create a new scope and pass the current value as a parameter.

Modern JavaScript with let and const largely eliminates this bug in loop contexts, but the underlying mechanism still trips developers up in React effects and event listeners.

## Stale Closures in React

This is where closures become a daily concern for React developers. When you define a function inside a component, it closes over the component's state and props at the time it was created. If state updates, the function still references the old value — unless React re-creates the function.

The linter rule exhaustive-deps exists to catch exactly this. When you use a stale value in an effect, the effect does not see the update. When you use a stale value in an event handler memoised with useCallback, the handler silently operates on outdated data.

The most common manifestation: a useEffect that should run when a piece of state changes, but the developer does not add that state to the dependency array. The effect runs once and captures the initial state. The state updates, but the effect never re-runs, so it continues to operate on the initial value.

## Closures as a Design Pattern

Once you understand closures, you can use them intentionally. The module pattern uses closures to create private state. A counter function that exposes increment, decrement, and value — but keeps the internal count variable inaccessible from outside — is a closure.

React hooks themselves are implemented with closures. The useState hook closes over the component's state in the React runtime. The hook can read and update that state because it has access to the closed-over reference, even though the hook itself is a plain function call.

Custom hooks are closures too. When you write useLocalStorage, it captures the key you pass it and closes over the useState setter. The hook "remembers" the key without you having to pass it again on every interaction.

## Memoisation With Closures

Memoisation — caching the result of expensive function calls — is naturally expressed with closures. A memoised function closes over a cache object. On each call, it checks the cache for the given arguments. If a cached result exists, it returns it; if not, it computes the result, stores it in the cache, and returns it.

The cache persists between calls because it is closed over. It is not a global variable — it is private to the memoised function. This is clean, composable design.

Treat every function you write as a potential closure. Ask: what variables does this function access? When were those variables bound? When might they change?

In React, pay attention to what your hooks and handlers capture. When the linter flags a missing dependency, take it seriously rather than suppressing the warning. The stale closure bug it is preventing is genuinely hard to debug.

## The Three-Tier Model

The most robust token architectures have three tiers. Global tokens at the top define the raw palette — every colour, spacing value, and type scale in the system. These are descriptive: --blue-500, --space-4, --text-lg. They do not carry semantic meaning.

Semantic tokens sit in the middle. They reference global tokens and carry meaning: --color-primary, --color-background, --color-text-default. The semantic tokens are what components reference. Global tokens exist only to be aliased into semantic ones.

Component tokens are the third tier — component-specific overrides that components expose as their styling API. --button-bg, --card-border-radius, --input-focus-color. These default to semantic tokens but can be overridden in context without touching the component's internal CSS.

If they reference --color-primary instead, you change the semantic token once and everything updates. This indirection feels unnecessary until the day you ship a white-label version of the product with a different colour palette, and you update one file instead of forty.

The same principle applies to spacing and typography. A component that uses --space-4 is coupled to a specific spacing scale. A component that uses --card-padding is decoupled — it can be adjusted independently without changing the global scale.

## Theming at Multiple Scopes

CSS custom properties inherit, which means you can override tokens at any scope. Global overrides on :root affect everything. Overrides on a specific component class affect only that component. Overrides on a data-theme attribute affect everything inside the themed subtree.

This means your theme system can be granular. A dark mode overrides semantic tokens at :root. A high-contrast mode overrides a subset of tokens. A component in a marketing section can define a different --color-primary without affecting the product UI. All of this is achievable without JavaScript, without runtime style injection, and without any build-time complexity.

## Documentation as Code

Naming tokens is documentation. --color-bg-elevated communicates intent — this is the background for elevated surfaces like cards and modals — far better than --gray-100, which requires the reader to look up what --gray-100 means.

The best token systems are self-explanatory. A new designer or developer should be able to look at the token list and understand the system's design intent without a separate spec document.

## Migration Is Inevitable

Plan for migration from the beginning. Every token system eventually needs to change — values shift, names improve, new categories emerge. A token system that is strictly organised and consistently applied is far easier to migrate than one that has grown organically.

The practical move: give every token a prefix that identifies its tier (--global-*, --color-*, --card-*). This makes automated codemods easier and makes the token's role in the system visible at a glance.

Token systems are long-lived infrastructure. The time you invest in their structure at the beginning pays compounding returns over the life of the product. Get the architecture right, document it clearly, and enforce it in code review. ## AnimatePresence for Exit Animations

The single most useful feature in Framer Motion is AnimatePresence. React does not let you animate elements out of the DOM — when you conditionally render something, it either is in the DOM or is not. AnimatePresence intercepts the unmount and holds the element in the DOM long enough for the exit animation to complete.

The pattern: wrap any conditionally rendered element in AnimatePresence, and add an exit prop to the motion component inside it. When the condition becomes false, Framer Motion runs the exit animation before removing the element.

This makes modal dismissals, dropdown closings, and notification dismissals feel smooth without any additional complexity. The animation is defined alongside the rest of the component's motion configuration — there is no separate unmount logic to manage.

## Layout Animations with layoutId

The layout prop on a motion component animates any change in the component's position or size, including changes caused by other elements moving. This is the feature that makes element rearrangements look polished — add a filter to a list and the remaining elements animate smoothly into their new positions.

layoutId is a shared key across motion components. When one component with a layoutId unmounts and another with the same layoutId mounts, Framer Motion animates the transition between them. This is the mechanism for "hero" transitions — the thumbnail from a grid that expands into a full-size detail view.

The implementation is simpler than it sounds: both the grid item and the detail view use the same layoutId. When you navigate from grid to detail, Framer Motion handles the position and size interpolation automatically.

## Gesture-Driven Motion

Framer Motion's gesture props — drag, whileHover, whileTap — are composable and independent. A draggable element with whileTap scaling and whileHover highlight just works. The variants are applied and reversed automatically.

The drag prop is particularly powerful because it handles the underlying pointer event complexity. A swipeable card stack — each card draggable, snapping back or flying away based on distance — is 30 lines of motion component configuration. The same behaviour in vanilla JS would be several hundred lines.

dragConstraints limits how far an element can be dragged. dragElastic controls how much the element overshoots the constraints. onDragEnd gives you the final velocity and offset for conditional logic — "if the card was dragged more than 100px, dismiss it."

## Stagger with Variants

The variants API lets you define animation states as named objects and apply them across a tree of motion components. The parent controls which variant is active; children inherit it.

The staggerChildren property on a parent variant staggers when each child begins its transition. This turns a list of items appearing simultaneously into a cascade. The amount of stagger — typically 50-100ms per item — is what makes list appearances feel deliberate rather than arbitrary.

The key detail: children must define the same variant names as the parent. When the parent enters the visible variant, children automatically transition to their visible variant, offset by the stagger delay.

## Performance Considerations

Framer Motion animations run on the compositor thread when they animate transform and opacity — the same properties that hardware-accelerated CSS animations use. When they animate layout (position, size), they run on the main thread and can cause layout thrashing.

For performance-critical animations, stick to transform and opacity. For layout animations, use the layout prop judiciously — it is powerful but has a higher cost. Measure with React DevTools Profiler and browser performance tools if animations are not hitting 60fps.

## useAnimation for Imperative Control

The useAnimation hook returns an animation controller that you can call .start(), .stop(), and .set() on. This is for cases where you need to trigger animations from logic rather than from component state — responding to a WebSocket event, sequencing complex animations, or syncing animation with audio.

The hook is complementary to the declarative variant system, not a replacement. Reach for it when the declarative approach requires complex conditional logic that obscures intent.

## A Note on Reduced Motion

Framer Motion respects prefers-reduced-motion through the useReducedMotion hook, which returns true when the user has requested reduced motion. Use it to conditionally set duration to 0 or replace movement-based animations with opacity-only equivalents.

This is not optional. Motion sickness is real, and some users have vestibular disorders that make layout animations genuinely uncomfortable. Every animation that moves elements spatially should have a fallback for reduced motion.
