Framer Motion: Practical Patterns for Real Apps

Framer Motion is more than a spring animation library. These patterns — shared layout animations, presence transitions, gesture-driven motion — are the ones I reach for repeatedly in production.

November 2, 20248 min read

Framer Motion has the best developer experience of any animation library I have used. The API is small, the defaults are sensible, and the integration with React is natural. But the patterns that make it genuinely powerful are not the ones you find in the introductory examples. This is a guide to the patterns I use in real production apps.

AnimatePresence for Exit Animations

The single most useful feature in Framer Motion is AnimatePresence. React does not let you animate elements out of the DOM — when you conditionally render something, it either is in the DOM or is not. AnimatePresence intercepts the unmount and holds the element in the DOM long enough for the exit animation to complete.

The pattern: wrap any conditionally rendered element in AnimatePresence, and add an exit prop to the motion component inside it. When the condition becomes false, Framer Motion runs the exit animation before removing the element.

This makes modal dismissals, dropdown closings, and notification dismissals feel smooth without any additional complexity. The animation is defined alongside the rest of the component's motion configuration — there is no separate unmount logic to manage.

Layout Animations with layoutId

The layout prop on a motion component animates any change in the component's position or size, including changes caused by other elements moving. This is the feature that makes element rearrangements look polished — add a filter to a list and the remaining elements animate smoothly into their new positions.

layoutId is a shared key across motion components. When one component with a layoutId unmounts and another with the same layoutId mounts, Framer Motion animates the transition between them. This is the mechanism for "hero" transitions — the thumbnail from a grid that expands into a full-size detail view.

The implementation is simpler than it sounds: both the grid item and the detail view use the same layoutId. When you navigate from grid to detail, Framer Motion handles the position and size interpolation automatically.

Gesture-Driven Motion

Framer Motion's gesture props — drag, whileHover, whileTap — are composable and independent. A draggable element with whileTap scaling and whileHover highlight just works. The variants are applied and reversed automatically.

The drag prop is particularly powerful because it handles the underlying pointer event complexity. A swipeable card stack — each card draggable, snapping back or flying away based on distance — is 30 lines of motion component configuration. The same behaviour in vanilla JS would be several hundred lines.

dragConstraints limits how far an element can be dragged. dragElastic controls how much the element overshoots the constraints. onDragEnd gives you the final velocity and offset for conditional logic — "if the card was dragged more than 100px, dismiss it."

Stagger with Variants

The variants API lets you define animation states as named objects and apply them across a tree of motion components. The parent controls which variant is active; children inherit it.

The staggerChildren property on a parent variant staggers when each child begins its transition. This turns a list of items appearing simultaneously into a cascade. The amount of stagger — typically 50-100ms per item — is what makes list appearances feel deliberate rather than arbitrary.

The key detail: children must define the same variant names as the parent. When the parent enters the visible variant, children automatically transition to their visible variant, offset by the stagger delay.

Performance Considerations

Framer Motion animations run on the compositor thread when they animate transform and opacity — the same properties that hardware-accelerated CSS animations use. When they animate layout (position, size), they run on the main thread and can cause layout thrashing.

For performance-critical animations, stick to transform and opacity. For layout animations, use the layout prop judiciously — it is powerful but has a higher cost. Measure with React DevTools Profiler and browser performance tools if animations are not hitting 60fps.

useAnimation for Imperative Control

The useAnimation hook returns an animation controller that you can call .start(), .stop(), and .set() on. This is for cases where you need to trigger animations from logic rather than from component state — responding to a WebSocket event, sequencing complex animations, or syncing animation with audio.

The hook is complementary to the declarative variant system, not a replacement. Reach for it when the declarative approach requires complex conditional logic that obscures intent.

A Note on Reduced Motion

Framer Motion respects prefers-reduced-motion through the useReducedMotion hook, which returns true when the user has requested reduced motion. Use it to conditionally set duration to 0 or replace movement-based animations with opacity-only equivalents.

This is not optional. Motion sickness is real, and some users have vestibular disorders that make layout animations genuinely uncomfortable. Every animation that moves elements spatially should have a fallback for reduced motion. Framer Motion makes this easy — use it.
