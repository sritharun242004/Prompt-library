The deployment strategies worth knowing

March 7, 20244 min read
devopsarchitecturetooling

Most deployment incidents are not caused by the code being wrong. They're caused by the code being right but the deployment strategy being wrong for the situation. A bug that would have been caught by a 5% canary rollout instead hits 100% of production at once. A schema migration that should have been decoupled from the application deploy happens simultaneously and takes down both.

Understanding deployment strategies is understanding risk management. The strategy you choose determines how much production traffic a bad deploy can affect before you catch it, and how reversible the mistake is.

Blue-green deployments

Blue-green is the most conceptually simple strategy: run two identical production environments, switch traffic between them. "Blue" is live; you deploy to "green," run your verification, then flip the load balancer. Rollback is instant — flip back to blue.

The catch is cost and statefulness. Running two full production environments is expensive. More importantly, if your application is stateful — sessions, queued jobs, websocket connections — the cutover creates discontinuities. Sessions on the old environment may be lost; in-flight requests get interrupted. Blue-green works best for stateless services where you can drain connections gracefully before the switch.

# Example GitHub Actions step for a blue-green deploy
- name: Deploy to green environment
  run: |
    kubectl apply -f k8s/green-deployment.yaml
    kubectl rollout status deployment/app-green

- name: Run smoke tests against green
  run: npm run test:smoke -- --env=green

- name: Switch traffic to green
  run: kubectl patch service app --patch '{"spec":{"selector":{"slot":"green"}}}'

This pattern is common in Kubernetes environments where you can label deployments and update the service selector atomically.

Canary releases

A canary release is a partial deployment: route a small percentage of production traffic to the new version while the rest continues to the old version. Monitor the new version. Gradually increase the percentage if metrics hold.

The advantage over blue-green is that canary tests real production traffic, not synthetic smoke tests. You'll catch bugs that only appear under real load patterns, real data shapes, or geographic distribution that your test environment doesn't replicate.

The configuration I use at Vercel for significant feature deploys:

  • 2% traffic for 30 minutes — catch hard crashes
  • 10% traffic for 2 hours — catch performance regressions
  • 25% traffic for 4 hours — catch edge cases that need volume to surface
  • 100% — full deploy

At each stage, an automated check compares error rates and P99 latency between the canary and the stable cohort. If the canary is worse by a configurable threshold, the deploy pauses and pages the on-call. This is not optional discipline — it's baked into the deployment tooling so it can't be skipped.

Feature flags

Feature flags are not strictly a deployment strategy, but they solve the problem that all other deployment strategies leave partially solved: the decoupling of deploy from release.

With feature flags, you merge code into main and deploy it to 100% of production — but the feature is disabled. When you're ready to release, you flip the flag. Rollback is not a code change; it's a config change that takes effect immediately without a deploy.

The operational advantage is enormous for teams that ship continuously. You can merge work in progress, do database migrations ahead of the feature that needs them, and coordinate multi-service changes without timing deploys across services.

The complexity cost is real: flag cleanup is a genuine discipline. A codebase with 40 active flags and no process for retiring them becomes unreadable. The rule I follow: every feature flag gets a removal ticket created at the time the flag is created, with an owner and a target date.

When to use which

The honest answer is that these strategies aren't mutually exclusive, and mature teams use all three for different purposes:

  • Blue-green for infrastructure changes (database upgrades, runtime upgrades, certificate rotations) where you need instant rollback and can tolerate the cost of parallel environments.
  • Canary for application deployments where you want production validation before full exposure.
  • Feature flags for product features where you want to decouple the technical deploy from the product release, or where you need to release to a subset of users.

The failure mode I see most often is using no strategy — every deploy goes to 100% of production simultaneously, rollback means a revert commit and another deploy cycle (5-15 minutes minimum), and incidents become catastrophic instead of contained. Any of these strategies is better than that.
