Building with Next.js App Router

December 1, 20244 min read
next.jsreactarchitecture

The App Router didn't just change the file-based routing API. It changed the conceptual model underneath all of Next.js. Understanding why certain things work the way they do — instead of just memorising the rules — is what separates engineers who can debug App Router issues from those who spend hours confused by them.

Layouts are persistent, pages are not

In the Pages Router, every page was a standalone unit. The layout was re-executed on each navigation. In the App Router, layouts are persistent across navigations within the same segment. Only the part of the tree that changes gets re-rendered.

This sounds like an implementation detail, but it has practical consequences. State inside a layout component persists across navigations. If you put a sidebar filter in a layout, the user's selection survives moving between pages nested under that layout. This is exactly what you'd want — and exactly what was painful to wire up manually before.

// app/dashboard/layout.tsx — persistent across all /dashboard/* routes
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div className="flex h-screen">
      <Sidebar />          {/* state persists between navigations */}
      <main>{children}</main>
    </div>
  )
}

Nested routing is compositional, not hierarchical

The App Router's (group) folders, parallel routes, and intercepting routes are a compositional system. You can display two different routes simultaneously in the same viewport. You can intercept a route to show a modal instead of navigating away. These aren't escape hatches — they're designed primitives.

Parallel routes use @slot naming conventions:

app/
  layout.tsx
  @modal/
    (.)photo/[id]/
      page.tsx       ← intercepts /photo/[id] to render as modal
  photo/
    [id]/
      page.tsx       ← renders normally on direct navigation

When a user clicks a photo, they see the modal. When they refresh or land directly on the URL, they see the full page. Same route, different presentation depending on how you arrived.

Data fetching collapses into the component

One of the most underappreciated things about the App Router is that fetch in Server Components is automatically deduplicated within a request. Two Server Components that both fetch the same URL will only trigger one network request. This makes it safe to fetch data close to where you use it without worrying about duplicate requests.

// Both of these call the same endpoint — only one fetch happens
async function UserAvatar({ id }: { id: string }) {
  const user = await fetch(`/api/users/${id}`).then(r => r.json())
  return <img src={user.avatar} alt={user.name} />
}

async function UserName({ id }: { id: string }) {
  const user = await fetch(`/api/users/${id}`).then(r => r.json())
  return <span>{user.name}</span>
}

Next.js extends the native fetch API with a cache layer. You can control caching per request, per route, or disable it entirely. The default is force-cache for static rendering, no-store for dynamic rendering.

Route handlers replace API routes

app/api/route.ts files replace pages/api/*.ts. The API is cleaner: you export named functions for each HTTP method.

// app/api/subscribe/route.ts
import { NextRequest, NextResponse } from 'next/server'

export async function POST(request: NextRequest) {
  const { email } = await request.json()
  // handle subscription
  return NextResponse.json({ success: true })
}

The Request and Response objects are the standard Web API — not Node.js IncomingMessage. This means route handlers run correctly at the edge without any changes.

The mental model that ties it together

The App Router is a tree of Server Components that can contain Client Component islands. Data flows down. Events bubble up from clients to server actions. Layouts cache, pages render fresh. Once this tree structure is clear in your head, the otherwise-confusing rules — why you can't use useRouter in a Server Component, why a layout renders before the page data is ready — all become obvious rather than arbitrary.

The payoff: pages that are fast by default, data fetching that's genuinely colocated, and a routing system that can express any layout pattern you'd want to build.
