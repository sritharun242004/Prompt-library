Documentation is a design problem

January 15, 20244 min read
writingcareertools

Good documentation is hard to write because the hardest part isn't the writing. It's the empathy. You need to understand exactly what your reader knows, what they're trying to do, and where their mental model will diverge from yours — and then structure a path through all of that without making them feel lost or talked down to.

Most documentation fails because it's written from the perspective of the implementor. The author knows the system deeply and writes down what they think is important about it — the design decisions, the internal architecture, the edge cases they're proud of handling. The reader, who is trying to accomplish a specific task, searches the page for the answer to their specific question and finds three paragraphs about motivation that don't tell them what to type.

The task-first model

Before writing any documentation, I write down the five to ten tasks that someone reading the documentation is most likely trying to accomplish. These tasks drive everything:

  • What they already know (assumptions I can make)
  • What they need to know to complete the task (the minimum necessary knowledge)
  • What order the steps need to happen in (often different from the logical architecture)
  • What can go wrong and what to do about it

A task is not "understand the authentication system." A task is "add OAuth login to an existing Next.js application that currently uses username/password." The difference matters because the second version tells me exactly what to include and what to leave out.

Code examples as first-class content

The worst kind of documentation example is generic. doSomething(value, options) teaches me the API surface, not how to use it for my purpose. The best kind is realistic: it uses real-sounding data, reflects actual usage patterns, and shows the code in context rather than in isolation.

At Stripe, we had a rule: every code example had to be copy-pasteable and runnable with minimal changes. "Minimal changes" meant only swapping in real secrets and real IDs — not restructuring the code or understanding how all the pieces connected before you could try it. This rule forced us to think about what the example was assuming the reader already had set up.

The second rule: every example had to work. This sounds obvious and it's perpetually violated. Documentation examples that don't work erode trust faster than having no examples at all. We enforced it by running all code examples in CI against the actual API. When the API changed, the examples broke in CI before they shipped.

Error messages as documentation

An often-overlooked form of documentation is the error message the user sees when something goes wrong. Error messages are documentation under pressure — the user is already frustrated, they're already stuck, and they need exactly the right information in exactly the right order.

A good error message does three things: names what went wrong, explains why (if non-obvious), and tells the user what to do next. "Invalid API key" fails all three. "The API key you provided is not valid for this environment. Test mode keys start with sk_test_; production keys start with sk_live_. Check that you're using the right key for the right environment." passes all three.

The habit I've developed: when I write a new error condition in code, I immediately write the error message as if I were a user encountering it for the first time with no knowledge of the implementation. The error message is part of the feature, not an afterthought.

The maintenance problem

Documentation that's out of date is worse than no documentation. It sends users down paths that don't work, erodes trust in the documentation system generally, and creates support burden as users ask for help with things the docs told them wrong.

The only reliable solution is coupling documentation to the code it documents. Docs that live in the same repository as the code, reviewed in the same pull request, with the same owners, get updated when the code changes. Docs that live in a separate system get updated when someone notices they're wrong — which is never as quickly as the code changes.

This is the structural reason to prefer inline documentation (JSDoc, rustdoc, typedoc) over separate documentation sites where possible: the proximity forces maintenance. When you're changing a function signature, the doc comment is right there. When the docs are in another system, they're someone else's problem.

The ideal is coupling that makes ignoring the documentation impossible — a CI check that fails if public API surface changes without a corresponding documentation change. That's what forces the discipline that prose reminders never do.
