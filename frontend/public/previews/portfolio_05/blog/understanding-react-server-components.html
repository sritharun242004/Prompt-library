Understanding React Server Components

September 18, 20244 min read
reactnext.jsarchitecture

React Server Components are easy to understand once you have the right mental model, and surprisingly hard to reason about without one. The key question to ask about any component isn't "where does it run?" — it's "who owns it?"

The ownership model

A Server Component is owned by the server. It runs once, at request time (or build time), and produces a description of the UI. It can read from databases, call internal APIs, access the filesystem, and use any server-side module. It cannot hold state, respond to events, or access browser APIs.

A Client Component is owned by the browser. It runs initially on the server (for server-side rendering), then hydrates and becomes interactive on the client. It can use useState, useEffect, event handlers, and browser APIs. It cannot directly access server-only resources.

The 'use client' directive marks where ownership transfers. Everything in a module marked 'use client' and everything it imports becomes part of the client bundle.

When to use Server Components

Use Server Components for:

Data fetching. A component that reads from a database or external API and renders the result — no interactivity needed — is a perfect Server Component. The data never touches the client. The render output is just HTML (or a serialised React tree).

async function PostList() {
  const posts = await db.posts.findMany({
    orderBy: { createdAt: 'desc' },
    take: 10,
  })

  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>
          <a href={`/blog/${post.slug}`}>{post.title}</a>
        </li>
      ))}
    </ul>
  )
}

Static content. Headers, footers, sidebars, marketing copy — anything that doesn't change based on user interaction.

Large dependencies. If you need a syntax highlighting library, a markdown parser, or a date formatting library, keeping it in a Server Component means it never ships to the browser.

When to use Client Components

Use Client Components for:

Interactivity. Anything that responds to user events: buttons that open modals, inputs that filter a list, toggles that change state.

Browser APIs. localStorage, navigator, window, IntersectionObserver, ResizeObserver.

Real-time data. WebSocket connections, subscriptions, polling.

The smallest possible Client Component surface area is ideal. Instead of making an entire page a Client Component because it has one interactive button, extract the button into a Client Component and keep the rest of the page as Server Components.

// page.tsx — Server Component
import { LikeButton } from './LikeButton'  // Client Component

async function PostPage({ params }: { params: { slug: string } }) {
  const post = await getPost(params.slug)

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
      <LikeButton postId={post.id} initialLikes={post.likes} />  {/* Client island */}
    </article>
  )
}

The serialisation constraint

Props passed from Server Components to Client Components must be serialisable. This means: strings, numbers, booleans, plain objects, arrays, and null. Not: functions, class instances, Dates (use ISO strings instead), Sets, Maps.

This constraint exists because the Server Component's output needs to be transmitted from server to client — it crosses a serialisation boundary. If you try to pass a function, React will throw an error at runtime.

The practical consequence: event handlers must be defined in Client Components. You can't pass an onClick from a Server Component.

The composition pattern that trips people up

Server Components can render Client Components. Client Components can render other Client Components. Client Components cannot render Server Components — but they can render Server Components if they receive them as the children prop.

// This works — children come from a Server Component higher in the tree
'use client'
function Modal({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(false)
  return open ? <dialog>{children}</dialog> : null
}

// In a Server Component:
<Modal>
  <DataFetchingServerComponent />  {/* This is fine */}
</Modal>

The children pattern is the key to maintaining a lean client bundle while composing Server and Client Components freely.

The bottom line

Treat Server Components as the default. Reach for 'use client' when you need interactivity, browser APIs, or real-time data. Keep Client Components small and leaf-like in the tree. This gives you the best of both worlds: server-side data access and client-side interactivity, with a minimal JavaScript payload.
