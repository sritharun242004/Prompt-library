Building a Design System from Scratch

November 28, 20244 min read
design-systemscssfrontendarchitecture

Most design systems fail not because the components are bad but because the system wasn't designed for adoption. A beautiful component library that nobody uses is a worse outcome than a scrappy one that the whole team reaches for first.

This is what I've learned building design systems at scale — not the right component API, but the decisions that determine whether a system survives first contact with a real product team.

Start with tokens, not components

Design tokens are the shared vocabulary between design and engineering. Colors, spacing, typography scales, border radii, shadow levels — everything that a designer specifies and an engineer implements.

Define tokens before you write any component code. Tokens are the contract.

// tokens.ts — the ground layer
export const tokens = {
  color: {
    neutral: {
      0: '#ffffff',
      100: '#f5f5f5',
      200: '#e5e5e5',
      500: '#737373',
      900: '#171717',
      950: '#0a0a0a',
    },
    accent: {
      500: '#6366f1',
      600: '#4f46e5',
    },
  },
  space: {
    1: '0.25rem',
    2: '0.5rem',
    3: '0.75rem',
    4: '1rem',
    6: '1.5rem',
    8: '2rem',
    12: '3rem',
    16: '4rem',
  },
  radius: {
    sm: '0.25rem',
    md: '0.375rem',
    lg: '0.5rem',
    full: '9999px',
  },
} as const

With Tailwind, map these tokens to your tailwind.config.ts theme extension so components use the semantic names rather than arbitrary values. A color that's neutral.500 in a token is text-neutral-500 in a component — the naming is consistent and searchable.

The component hierarchy that works

Organize components in three layers:

Primitives: Box, Text, Stack, Cluster. These are layout and typography wrappers with no visual opinion. They implement the spacing and typography tokens.

Base components: Button, Input, Select, Card, Badge. These have visual opinions but no domain logic. They accept variant and size props, not business logic props.

Composed components: SearchInput (Input + Button), FormField (Input + Label + Error), ProductCard (Card + Image + Text). These are specific to your product. They live in the product codebase, not the design system package.

The mistake most teams make is putting composed components in the design system. Composed components are opinionated about product structure. When they're in the shared system, they're hard to update without breaking consumers.

The Button component, done right

A Button that handles every use case correctly is harder than it looks:

import { forwardRef } from 'react'
import { Slot } from '@radix-ui/react-slot'
import { cn } from '@/lib/cn'

interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'solid' | 'outline' | 'ghost'
  size?: 'sm' | 'md' | 'lg'
  asChild?: boolean   // render as a different element (e.g. Link)
  loading?: boolean
}

const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ variant = 'solid', size = 'md', asChild, loading, children, className, ...props }, ref) => {
    const Comp = asChild ? Slot : 'button'
    return (
      <Comp
        ref={ref}
        disabled={loading || props.disabled}
        className={cn(buttonVariants({ variant, size }), className)}
        {...props}
      >
        {loading ? <Spinner /> : children}
      </Comp>
    )
  }
)

The asChild pattern from Radix UI is essential. It lets you render <Button asChild><Link href="/...">Click</Link></Button> and get a proper <a> tag with all the button styles — no wrapping div, no accessibility issues.

Documentation is half the product

A component library without documentation has a 50% adoption rate at best. Engineers who can't figure out how to use a component in 60 seconds reach for a custom implementation instead.

Good documentation for each component needs:

  1. A live interactive example (not a screenshot)
  2. Every prop, with types and defaults
  3. The variants and when to use each
  4. Accessibility notes if relevant
  5. A "don't do this" example for the common misuse

Storybook is the standard, and it's good enough. The key is making documentation a first-class deliverable — not something that gets added after the component is "done."

Versioning and migration

Use semantic versioning. Breaking changes go in major versions. New features in minor. Fixes in patch.

Write automated codemods for breaking changes when possible. A codemod that migrates a prop rename is more likely to be adopted than written migration instructions. jscodeshift is the standard tool.

The design system's job is to make building product faster, not to enforce compliance. The team that uses your system because it's genuinely better is more valuable than the team that uses it because they're required to.
