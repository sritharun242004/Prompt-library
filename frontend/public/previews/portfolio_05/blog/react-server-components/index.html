React Server Components are not what you think

November 12, 20244 min read
reactnext.jsarchitecture

Server Components introduce a boundary that most tutorials gloss over. The marketing says "run components on the server," and technically that's true, but the real shift is about where the data lives relative to the render. Once you understand that, the otherwise strange rules — why you can't use useState, why async/await just works, why importing a database driver is fine — all snap into place.

The old model: components are functions that return UI, and they run wherever they're evaluated. You fetch data separately (in getServerSideProps, in a useEffect, in a data-fetching library), pass it as props, and the component doesn't know or care where it came from. The component's job is pure transformation: data in, JSX out.

The new model splits that contract. Server Components are functions that run exactly once, on the server, at request time (or build time for static). They have direct access to the data layer. Client Components are functions that run on the client and can re-run as state changes. The interesting constraint: a Server Component can render a Client Component, but a Client Component cannot render a Server Component. This isn't a limitation — it's a deliberate boundary that makes the split coherent.

Why async/await works in RSC

In a traditional React component, side effects during render are forbidden because the component might run multiple times, concurrently, in different contexts. That's why useState, useEffect, and friends exist — they're escape hatches that let React manage the timing.

Server Components don't have that problem. They run once, in a controlled environment, where React owns the scheduling. So async/await is simply... fetch. No wrapper needed. The component suspends while the awaited promise resolves, React handles that at the boundary, and the rendered output arrives fully formed.

// This is a Server Component — no "use client" directive
async function PostPage({ params }: { params: { slug: string } }) {
  const post = await db.posts.findUnique({ where: { slug: params.slug } })
  if (!post) notFound()

  return (
    <article>
      <h1>{post.title}</h1>
      <Prose content={post.body} />
    </article>
  )
}

That database call runs on the server. The query result never touches the browser. The component renders to HTML (or a serialisable React tree for client-side navigation) and that's what the browser receives.

The boundary is architectural, not syntactic

The 'use client' directive is the most confusing part of the API because it looks like it marks a component as "running on the client." It doesn't. It marks the module boundary — the point where the component and everything it imports moves from the server graph to the client graph.

This means: a module with 'use client' at the top will be bundled into the client JavaScript. Its imports will also be pulled into the client bundle. Nothing above that boundary is included — Server Components higher in the tree can pass serialisable props down to it, but they can't pass functions, class instances, or anything that can't be serialised over the wire.

If you internalize this, the rules stop feeling arbitrary. You can't call useState in a Server Component because state is a client concept — there's no mechanism to persist it between the one render the Server Component gets. You can't pass a callback from a Server Component to a Client Component because a function is not serialisable. You can pass children as JSX because React serialises the tree before sending it.

What this changes about how you think about data

The biggest practical shift: colocation becomes cheap again. Before RSC, fetching data close to where it's used meant doing it client-side (slow — waterfall network requests from the browser) or hoisting it to getServerSideProps (awkward — forces you to think in page-level data dependencies). Neither feels like it belongs to the component that needs the data.

With Server Components, you fetch exactly what you need, exactly where you need it, and it costs nothing in client bundle size. A <UserAvatar userId={id} /> Server Component can hit the database directly. It's not in the client bundle. The data never leaks to the browser. And you can compose it anywhere in the tree without wiring up props.

This is the mental model the docs skip. The docs tell you what RSC does. This is why it's structured the way it is.
