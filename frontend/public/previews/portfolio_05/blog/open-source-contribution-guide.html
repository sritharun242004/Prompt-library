Open Source Contribution Guide

April 5, 20244 min read
open-sourcecareercommunity

Most advice about contributing to open source focuses on the mechanics: fork the repo, make a branch, open a PR. The mechanics are the easy part. The hard part is finding the right entry point, doing work that maintainers will accept, and not losing momentum before your first contribution lands.

Here's what actually works.

Start smaller than you think

The instinct is to find a big impactful feature and implement it. This is how you get a PR open for six months that never merges because it has architectural implications the maintainers want to think through before committing to.

Better approach: find something small and obviously correct. Documentation improvements, test coverage for edge cases, fixing a failing test in CI, adding a missing type definition. These are low-risk for maintainers to review and approve. They also teach you how the project is structured, how maintainers communicate, and what their standards are — all of which makes your next contribution better.

Find projects where you're already a user

You'll do better work on software you actually use. You already know where the documentation is confusing, which error messages are unhelpful, which edge cases the library doesn't handle. These are the issues worth fixing.

Open your package.json or whatever dependency file your language uses and pick three libraries you depend on. Check their issue trackers. Look for issues tagged good first issue or help wanted. Read a few to understand the project's conventions before picking one.

The first thing to do before writing any code

Read the contributing guide. If there isn't one, read ten recent merged PRs to understand the commit message format, the PR template, the code style, and how detailed the review conversation tends to be.

The fastest path to an ignored PR is ignoring the project's conventions. Maintainers are volunteers. They have standards because those standards reduce the maintenance burden. Respecting them signals that you understand this.

How to write a PR description that gets reviewed

A PR description should answer three questions:

  1. What problem does this solve? (Link to the issue if one exists)
  2. How did you solve it?
  3. What did you test?

If the change touches something tricky, explain the tradeoff. Don't make the reviewer reverse-engineer your reasoning from the diff. Maintainers are busy; the easier your PR is to review, the sooner it gets reviewed.

## What

Fixes #1234 — `parse()` throws an unhandled error when the input contains
a null byte. Returns a `ParseError` instead.

## How

Added a null byte check before the main parsing loop. The check is O(n)
but so is the parse itself, so this doesn't change the complexity.

## Test

Added a test case in `src/__tests__/parse.test.ts` with a null byte in
the middle of a valid input string and at the end.

Responding to review feedback

The most important thing: don't take review feedback personally. Maintainers review dozens of PRs. Their comments are about the code, not about you.

Respond to every comment — even the ones you disagree with. If you disagree, explain why. Most maintainers respect pushback that's well-reasoned. What they don't want is silence (did you see the comment?) or immediate capitulation on things you have a strong opinion about.

If you make changes in response to feedback, don't force-push a rebase. Add new commits. The reviewer wants to see what changed since their last review; squashing the history makes that harder.

After it merges

Celebrate, then do it again. The second contribution to a project is much easier than the first — you know the structure, you know the maintainers' style, and your name is recognized. The compounding returns on consistent contribution to one project are much higher than spreading effort across many projects.

Some of the most influential contributors I've seen didn't work on the "exciting" features. They triaged issues, improved tests, and wrote documentation over years. That steady work is what sustains a project.
