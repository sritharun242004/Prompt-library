Web Performance Fundamentals

February 28, 20244 min read
performancewebfrontend

Core Web Vitals are Google's attempt to put numbers on user experience. Understanding what those numbers measure — not just the thresholds — is what allows you to improve them deliberately rather than by guessing.

LCP: Largest Contentful Paint

LCP measures when the largest visible element in the viewport finishes rendering. For most pages, this is either a hero image or the main heading.

The 2.5-second threshold reflects research on when users perceive a page as "loaded." Below 2.5 seconds, most users continue waiting. Above 4 seconds, significant abandonment begins.

What moves LCP:

Resource discovery. If the browser discovers the hero image in CSS or via JavaScript, it can't start downloading it until those resources are parsed. Put critical images directly in the HTML with <img> tags (not CSS backgrounds) and use fetchpriority="high" on the LCP image:

<img
  src="/hero.jpg"
  alt="Hero"
  fetchpriority="high"
  loading="eager"
  width="1200"
  height="630"
/>

Server response time. TTFB (Time to First Byte) is the ceiling for everything else. A slow server means a slow LCP. Optimize your database queries, use CDN caching for static content, and consider edge functions for personalized pages.

Render-blocking resources. CSS blocks rendering. JavaScript without async or defer blocks HTML parsing. Audit your <head> with Chrome's Coverage tool — you likely have CSS that's loading styles for components not on the current page.

CLS: Cumulative Layout Shift

CLS measures the total movement of visible elements during the page lifetime. A score above 0.1 is noticeable; above 0.25 is disruptive.

The most common causes:

Images without dimensions. When the browser doesn't know an image's size before it loads, it allocates no space for it. When it loads, everything below shifts down.

<!-- Bad: no dimensions -->
<img src="/photo.jpg" alt="photo">

<!-- Good: explicit dimensions -->
<img src="/photo.jpg" alt="photo" width="400" height="300">

Fonts with different metrics than fallbacks. System fonts and web fonts have different character widths. When a web font loads, text reflows. Use font-display: optional if you're okay with the system font, or use size-adjust in @font-face to match the fallback metrics:

@font-face {
  font-family: 'CustomFont';
  src: url('/font.woff2') format('woff2');
  font-display: swap;
  size-adjust: 105%;  /* match fallback width */
}

Dynamically injected banners. Cookie notices, newsletter prompts, and A/B tested sections that appear after initial render all cause shift. Reserve space for them, or animate them from the bottom of the viewport rather than pushing content down.

INP: Interaction to Next Paint

INP replaced FID in March 2024. Where FID only measured the first interaction, INP measures the worst interaction throughout the session. The threshold is 200ms for "good."

INP is a CPU problem. Long tasks on the main thread delay the browser's ability to respond to user input. The fix is breaking up long tasks:

// Bad: one long synchronous task
function processLargeList(items: Item[]) {
  items.forEach(item => expensiveOperation(item))
}

// Better: yield to the main thread between chunks
async function processLargeList(items: Item[]) {
  const chunkSize = 50
  for (let i = 0; i < items.length; i += chunkSize) {
    const chunk = items.slice(i, i + chunkSize)
    chunk.forEach(item => expensiveOperation(item))
    await scheduler.yield()  // give the browser a chance to paint
  }
}

The measurement workflow

Field data (real user measurements from CrUX) is the source of truth. Lab data (Lighthouse, WebPageTest) is for debugging. They'll often disagree because lab data runs on a fixed device and network; field data reflects your actual users' devices and conditions.

Use Lighthouse to find regressions in CI. Use the Chrome DevTools Performance panel to diagnose them. Use Search Console's Core Web Vitals report to understand field performance across pages.

The goal is never a specific number — it's building the feedback loops that let you catch regressions before they ship and diagnose them when they do.
