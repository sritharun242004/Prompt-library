My Development Setup in 2024

October 20, 20244 min read
toolsproductivitysetup

I get asked about my setup more than anything else. This is the authoritative answer — every tool, every configuration decision, and why I made it. I update this when something changes meaningfully.

Terminal: Ghostty

I switched to Ghostty when it launched and haven't looked back. Native GPU rendering means scrolling through long build logs feels frictionless in a way that iTerm2 never quite managed. The configuration is a single plain-text file with no XML, no JSON, just key = value pairs.

The most important terminal setting I have:

font-family = "Iosevka Term"
font-size = 14
theme = "GruvboxDark"
shell-integration = detect
cursor-style = bar

I use zsh with a minimal prompt — no Oh My Zsh, just a few lines of configuration. The prompt shows: current directory, git branch if in a repo, and a red indicator if the last command failed. Nothing else.

Editor: Neovim (with LazyVim)

I moved from VS Code to Neovim three years ago and I've never measured a productivity difference. What I have measured is fewer RSI flare-ups, because keeping my hands on the home row matters when you're typing eight hours a day.

LazyVim handles the plugin management and gives you a reasonable baseline configuration. I override very little:

-- ~/.config/nvim/lua/config/options.lua
vim.opt.relativenumber = true
vim.opt.scrolloff = 8
vim.opt.signcolumn = "yes"
vim.opt.wrap = false
vim.opt.conceallevel = 0   -- show markdown syntax, don't hide it

The plugins I actually use:

  • nvim-lspconfig + mason.nvim — language server management
  • nvim-cmp — completion (connected to LSP + snippets)
  • telescope.nvim — fuzzy finding (files, grep, git, LSP references)
  • gitsigns.nvim — git blame and hunk navigation in the gutter
  • conform.nvim — formatting on save (prettier, eslint, rustfmt)

Version management: mise

I used nvm for years and fnm briefly. mise replaces both and also handles Python, Ruby, Go, and any other runtime that has ASDF plugins. One tool, one .tool-versions file at the repo root, versions switch automatically when you cd into the directory.

# .tool-versions at repo root
nodejs 22.4.0
pnpm 9.4.0

Package manager: pnpm

The disk space savings alone justify switching. pnpm uses a content-addressable store — packages are installed once globally and hard-linked into each project's node_modules. A monorepo with ten packages that all depend on React doesn't have ten copies of React on disk.

The lockfile is more deterministic than npm's, and the strict mode (default) catches packages that accidentally depend on unlisted transitive dependencies — a class of bug that causes mysterious failures in CI but not locally.

Git configuration

Three settings that changed how I use git:

git config --global push.autoSetupRemote true   # no more --set-upstream
git config --global pull.rebase true             # rebase on pull
git config --global rerere.enabled true          # remember conflict resolutions

rerere is the most underrated one. If you resolve a merge conflict and then need to rebase over the same commit, git remembers your resolution and applies it automatically. Without this, rebasing a long-lived branch against main is painful.

Browser: Arc

Arc's sidebar model clicked for me in a way I didn't expect. Spaces let you keep work and personal tabs genuinely separate — not in different windows, but in different contexts that feel meaningfully distinct. The command bar (Cmd+T) handles navigation, new tabs, and history search in one place.

For development specifically: the devtools panel arrangement I settled on is Sources on the left half, Console on the right half, with the Network panel accessible via keyboard shortcut. I use the Performance panel weekly; the Memory panel rarely but when I need it I really need it.

The things I tried and dropped

Worth mentioning what didn't stick: Warp (too opinionated about the UI, the AI features feel intrusive), Zed (promising but the Neovim bindings aren't there yet), tmux (I tried three times; the cognitive overhead of pane management isn't worth it for the way I work).

The best setup is the one you understand well enough to fix when it breaks.
