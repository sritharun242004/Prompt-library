How to set a performance budget that your team will actually respect

September 4, 20244 min read
performancewebtooling

Performance budgets fail for two reasons: they measure the wrong thing, or they're checked manually. Fix both and you have a tool that actually changes how a team ships.

The wrong thing to measure is bundle size. Bundle size is a proxy metric — what you care about is Time to Interactive, or Largest Contentful Paint, or whatever the user-facing metric is for your product. A team that's focused on keeping bundle.js under 300kb will make perverse decisions: they'll tree-shake libraries that could have been lazy-loaded more effectively, or they'll move features out of the critical path in ways that technically pass the budget but make the page feel slower.

Measure what the user experiences. Use Lighthouse CI in your build pipeline. Set a budget on LCP (say, under 2.5 seconds on a simulated 4G mobile connection) and block merges that regress it. When the CI failure message says "LCP regressed from 2.1s to 3.4s on mobile," the team understands exactly what broke and why it matters.

Making budgets automatic

The discipline that makes performance stick is the same discipline that makes code quality stick: automation. A performance budget that lives in a Notion doc and requires someone to check a dashboard after deploys will be ignored. Not because people don't care — because they have other things to care about.

Lighthouse CI integrates directly with GitHub Actions and pulls request checks. Configuration is a single JSON file at the repo root:

{
  "ci": {
    "collect": {
      "url": ["http://localhost:3000", "http://localhost:3000/blog"],
      "numberOfRuns": 3
    },
    "assert": {
      "assertions": {
        "largest-contentful-paint": ["error", { "maxNumericValue": 2500 }],
        "total-blocking-time": ["warn", { "maxNumericValue": 300 }],
        "cumulative-layout-shift": ["error", { "maxNumericValue": 0.1 }]
      }
    }
  }
}

error blocks the merge. warn posts a comment but allows it. Use error sparingly — only for the metrics that genuinely determine whether the page is usable. Use warn for everything else. If everything is an error, engineers start treating failures as bureaucracy and look for ways around the check.

The budget conversation to have before you ship

Before adding a Lighthouse budget to a project, have the team agree on a target median and a regression threshold — separately. The target is aspirational: "We want p50 LCP under 2 seconds." The threshold is the floor: "We will block any deploy that takes LCP above 3.5 seconds." These are different numbers, and conflating them is how budgets become demoralising.

A regression threshold gives the team room to ship incrementally. If today's p50 LCP is 2.8 seconds, setting the threshold at 2.9 seconds means you're not blocking work — you're preventing things from getting worse while the team works on improving them. Once you've hit 2 seconds, tighten the threshold.

The goal is a feedback loop, not a gate. Gates create resentment. Feedback loops create understanding.

What to do when you fail the budget

When a PR fails the Lighthouse check, the first instinct is to look at what changed in that PR. That's usually the right instinct, but not always. Performance regressions are sometimes latent — a previous PR laid the groundwork (added a large dependency, changed a render order), and the current PR crossed the threshold.

Use the performance timeline in Chrome DevTools to understand when things happen, not just what happens. The waterfall view will tell you if you have a chain of blocking requests. The main thread activity panel will tell you if JavaScript is occupying the thread during the critical paint window. The layout shifts section will identify elements that move after the first paint.

Most regressions I've seen fall into three categories: a new third-party script added without async/defer, a lazy-loaded image that turned out to be above the fold, or a data fetch that moved from build time to request time without anyone noticing. The budget catches it. DevTools tells you where to look.
