TypeScript patterns I use every day

July 22, 20244 min read
typescriptjavascripttools

TypeScript's type system is expressive enough to eliminate entire categories of bugs, but most of the patterns that do real work aren't in the official docs. Here are the ones I reach for daily.

Branded types for IDs

The most common runtime bug in any typed system: passing a UserId where a PostId was expected. TypeScript's structural type system won't catch this — both are string, so they're assignable to each other. Branded types fix it with zero runtime cost.

type Brand<T, B extends string> = T & { readonly __brand: B }

type UserId = Brand<string, 'UserId'>
type PostId = Brand<string, 'PostId'>

function getUser(id: UserId): User { /* ... */ }
function getPost(id: PostId): Post { /* ... */ }

const userId = 'abc123' as UserId
const postId = 'xyz789' as PostId

getUser(postId) // Type error — correct
getPost(userId) // Type error — correct

The as UserId cast is the only place you create a branded value. In practice, you do this in a constructor or a parse function at the boundary where the value enters the system (a database row, an API response). Everywhere else, the type flows without casts.

Discriminated unions for state machines

React state that uses multiple booleans to represent a state machine (isLoading, isError, isSuccess) is a smell. You can represent states that shouldn't coexist simultaneously as the same state — isLoading: true, isSuccess: true is structurally valid but semantically impossible.

Discriminated unions make the impossible states unrepresentable:

type FetchState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: Error }

function Component() {
  const [state, setState] = useState<FetchState<Post>>({ status: 'idle' })

  if (state.status === 'loading') return <Spinner />
  if (state.status === 'error') return <ErrorMessage error={state.error} />
  if (state.status === 'success') return <PostView post={state.data} />
  return <IdleView />
}

Inside each branch, TypeScript narrows the type correctly. state.error is only available in the error branch. state.data is only available in the success branch. No optional chaining needed. No runtime null checks.

satisfies for config objects

The satisfies operator was added in TypeScript 4.9 and it's one of the most underused features. It lets you check a value against a type without widening the type.

const routes = {
  home: '/',
  blog: '/blog',
  work: '/work',
} satisfies Record<string, string>

// 'home' | 'blog' | 'work' — NOT string
type RouteName = keyof typeof routes

Without satisfies, you'd either annotate the type (const routes: Record<string, string>) and lose the key literal types, or leave it unannotated and lose the type check. satisfies gives you both.

Template literal types for event names

When an API accepts string event names and you want completion without an enum, template literal types create a constrained string space:

type EventName = `on${Capitalize<string>}`

function addEventListener(name: EventName, handler: () => void) { /* ... */ }

addEventListener('onClick', handler)   // fine
addEventListener('onHover', handler)   // fine
addEventListener('click', handler)     // error — doesn't start with 'on'

This pattern is particularly useful for plugin systems and pub/sub APIs where the event namespace is open but you want to enforce a naming convention.

infer for extracting types from generics

infer is the escape hatch for complex conditional types, and once you understand it, you'll see uses for it everywhere:

type UnwrapPromise<T> = T extends Promise<infer U> ? U : T
type UnwrapArray<T> = T extends Array<infer U> ? U : T

type A = UnwrapPromise<Promise<string>>  // string
type B = UnwrapArray<number[]>            // number
type C = UnwrapPromise<string>            // string (passthrough)

The built-in Awaited<T> utility type uses exactly this pattern. Understanding infer means you can write your own utilities when the built-ins don't quite fit.

These patterns don't add complexity for its own sake. Each one encodes a constraint that would otherwise live in a comment, a test, or a runtime check. TypeScript earns its complexity when it makes incorrect programs unrepresentable.
