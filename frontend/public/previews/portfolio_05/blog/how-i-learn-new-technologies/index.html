How I Learn New Technologies

June 10, 20244 min read
careerlearningproductivity

After a decade of keeping up with a field that never stops moving, I've refined a process. This isn't about speed-running tutorials. It's about building genuine understanding that transfers to the next thing you'll have to learn.

The funnel: wide to narrow

I start every new technology the same way: a 20-minute skim of the official documentation's "Why" page (if it exists), the README, and one introductory blog post from someone who has been using it for at least a year. Not a tutorial. A practitioner's retrospective.

This gives me the mental scaffolding before I see any code. What problem does this solve? What does it not try to solve? Who built it and why? A technology's design philosophy reveals itself in the constraints it imposes. Knowing those constraints in advance makes every subsequent example make sense instead of seeming arbitrary.

Build something that breaks

The second step is to build something real — not a to-do app, but something small enough to finish in a weekend and real enough to encounter genuine constraints.

When I learned Rust, I built a command-line tool that parsed a binary log format. When I learned tRPC, I added it to an existing project that already had REST endpoints so I could compare them in context. When I learned Temporal (the workflow engine), I ported a background job system I'd written in Redis to see how the abstractions mapped.

The choice of project is important: it should be in a domain you already understand, so you're not learning the problem and the tool simultaneously. If you're learning a new framework, build something in an area you know well enough to have opinions about the right behavior.

Let yourself be confused, then dig

When something doesn't work the way I expected, I resist the urge to immediately search for the answer. I sit with the confusion for ten minutes and try to articulate exactly what my prediction was and why it was wrong.

This sounds inefficient. It's the opposite. Most "I just can't get this to work" frustration comes from a wrong mental model that a Stack Overflow answer will patch without fixing. Ten minutes of deliberate confusion-articulation usually surfaces the model error. Once I know what I misunderstood, I search for an explanation of that specific thing — not a fix for the symptom.

Read the source

Every mature open-source project has readable source code if you know where to look. I try to read at least one relevant piece of the source for every major technology I adopt: how Next.js resolves routes, how Prisma generates types, how Zod traverses a schema.

The source is the ground truth. Documentation can be incomplete or outdated. The source is always correct.

I use a two-step process: find the feature I care about by searching the test suite (tests show exactly how something is supposed to behave), then trace the implementation from there.

Teach it before you feel ready

The best learning accelerator is explaining something to someone else before you're confident you understand it. Write a post, answer a question in the community Discord, pair with a colleague. The act of explaining exposes every gap in your understanding — gaps you wouldn't discover by reading more documentation.

I drafted this post while I was learning Temporal, and writing it forced me to realize I didn't actually understand how Temporal handles workflow versioning. I had glossed over it. Writing the sentence revealed the gap; the gap sent me back to the documentation; the documentation made the prior section I'd half-understood click into place.

The meta-skill

None of this is about learning faster. It's about retaining what you learn and building a foundation that makes the next thing cheaper to learn. The goal is a compounding knowledge base, not a race through a checklist of technologies.

The engineers I've seen learn the fastest aren't the ones who finish the most tutorials. They're the ones who ask the most precise questions — and precision comes from slowing down, building mental models, and refusing to move on while something is still fuzzy.
