Building Accessible Design Tokens

Design tokens are not just a theming convenience — they are the right abstraction layer for encoding accessibility constraints into your system.

AccessibilityDesign SystemsTokens

Design tokens are usually pitched as a theming mechanism: change a value in one place, update everywhere. That is true, but it undersells their potential. The token layer is the ideal place to encode accessibility constraints — colour contrast ratios, minimum touch target sizes, focus ring specifications — as first-class system requirements rather than checklist items.

The most impactful accessibility token to define is your focus ring. Most design systems treat focus styles as an afterthought — a global outline reset followed by custom per-component overrides. The better model is a focus token: a CSS custom property that specifies the ring color, width, offset, and border-radius, applied consistently via a single rule. When you need to update it (for high-contrast mode, for a dark theme, for a specific user preference), you update one token.

Colour contrast tokens deserve their own architecture. Instead of storing colours as atomic values (blue-500: #3B5BDB), store them as semantic pairs: foreground-on-card, foreground-on-accent, border-interactive. The pair abstraction forces you to think about contrast at the system level rather than case-by-case. It also makes dark mode dramatically simpler — you swap the pairs, not individual atomic values.

The most advanced token accessibility pattern is intent-based colour: instead of text-primary and text-secondary, use text-body, text-label, text-caption, text-disabled. Each intent maps to a minimum contrast level. This makes it structurally impossible to use a low-contrast token in a high-importance context — the system rejects it because the token name does not match the use case.
