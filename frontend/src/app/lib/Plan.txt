 Workstream C — Engine into Builder + Improver (#4)

 Goal: after Claude generates/improves, run the lock engine on the output and return a lock layer + negative locks + validation,
 surfaced in the UI.

 Backend — routes/builder.ts:
 - Keep the existing Claude generation. Add an optional category?: string to the request (the Builder UI sends one of the 7 image
 categories; default = infer from family/idea).
 - After generation, build an in-memory PromptRecord ({ id, title: idea, category, description: generatedText, platforms: {
 [platform]: generatedText } }) and call assembleFromRecord(record, platform).
 - Extend the response with lockSection, negativeLocks, validation, categoryId. Image-only (video/text/content skip the lock step —
 engine handles only image categories; passthrough returns empty locks).

 Backend — routes/improver.ts:
 - Keep the Claude improvement. Detect category from the improved text via resolveCategoryId on extracted cues (or a lightweight
 category hint param); build a PromptRecord from the improved text and call assembleFromRecord.
 - Add lockSection, negativeLocks, validation, categoryId to the response; optionally add lock-related entries to the changes array
 ("Added Lighting lock", etc.).

 Frontend — pages/Builder.tsx, pages/Improver.tsx, lib/api.ts:
 - Extend BuilderResult / ImproverResult types with the new fields.
 - Add a collapsible "Lock Layer" panel (renders lockSection as LOCK 01 - LABEL: value) and a "Negative Locks" list, plus a small
 validation badge (valid / N missing). Builder gets a category selector (7 categories) feeding the new category param.

 Reuses: assembleFromRecord, formatLockSection, resolveCategoryId, registry — no engine changes needed.

 ---
 Suggested execution order

 C (engine — highest product value, self-contained, testable offline) → A (UI/UX polish) → B (Guide). Each committed separately on
 feat/prompt-intelligence-engine (or a new branch per the user's preference).

 Verification

 - Backend (C): cd backend && npm run build. Unit-style check: a small script (like engine-smoke.ts) feeding a sample generated
 prompt through the new builder/improver post-processing to confirm lockSection/negatives populate. Live: with ANTHROPIC_API_KEY set,
 POST /api/builder/generate and /api/improver/improve return the new fields.
 - Frontend (A, B, C): cd frontend && npm run build (must stay green). Run npm run dev and manually verify: Home palette matches app,
 no broken "coming soon", Builder/Improver show the lock panel, Guide deep-links work, keyboard nav on cards.
 - Commit each workstream separately; rebuild after each.

 Risks / notes

 - Builder/Improver live testing needs ANTHROPIC_API_KEY (already used by those routes). The engine post-processing itself is pure
 and testable offline.
 - Category inference for Improver is heuristic; where it can't resolve, it gracefully returns no lock layer (passthrough) rather
 than a wrong one.
 - "General polish" is scoped to the bounded list above to avoid an unbounded redesign; larger redesigns can be a follow-up.
